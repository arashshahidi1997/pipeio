## `pipeio_run` resolves `snakemake` from a shared conda env instead of the project's pixi env → silent failure of scheduled agents

### Symptom

Scheduled agents (running inside the MCP sandbox) cannot run Snakemake
pipelines in projects that have migrated to pixi. They hit
`ModuleNotFoundError: No module named 'snakebids'` at Snakefile import
time because `pipeio_run` invokes
`/storage/share/python/environments/Anaconda3/envs/rag/bin/snakemake`
rather than the project-local `.pixi/envs/default/bin/snakemake`.

This is the root cause of a batch of rubber-stamped failures in gecog
today (2026-04-23):

- `preprocess_motion` retry — correctly stopped and filed issue
  [`docs/log/issue/issue-arash-20260423-021218-337742.md`](../../../../gecog/docs/log/issue/issue-arash-20260423-021218-337742.md)
- `factor_analysis` end-to-end run — no output produced, completed
  exit_code=0
- `brainstate` scaffold — skipped the datalad subdataset step, likely
  hit same block

Combined with the existing pipeio exit-code-masking bug
([issue-arash-20260421-210505-291234.md](docs/log/issue/issue-arash-20260421-210505-291234.md)),
these scheduled agents report `exit_code=0` and `status=ok` while
actually doing nothing.

### Repro

From inside a scheduled-agent sandbox in gecog:

```
pipeio_run(flow="preprocess_motion", targets=["test"], dryrun=True)
```

Log shows Snakefile line 5 crashes on `from snakebids import ...`.
Direct `which snakemake` in the sandbox returns the rag conda env path.
`.pixi/envs/default/bin/snakemake` works but isn't on PATH or selected
by pipeio.

### Where it broke

- Interactive `pipeio_run` calls from the primary session worked earlier
  today (`run_id=ed227976` on spectrogram) because our local PATH was
  different (we had activated pixi or similar).
- The previous preprocess_ieeg run (`run_id=429f1ea8`, 2026-04-21) also
  used `conda run snakemake`, so the rag env had `snakebids` installed
  **at that time** but doesn't now. Whoever owns rag uninstalled it,
  breaking everyone silently.

### Suggested fix (pick one, or both)

1. **Respect the project's pixi env.** If `<project>/pixi.toml` exists,
   invoke `pixi run --manifest-path <project>/pixi.toml snakemake ...`
   instead of the global `snakemake` binary. This matches the declared
   project env and is robust to shared-conda drift.
2. **Expose `snakemake_cmd` via the MCP tool schema.** Let the caller
   override the binary per invocation (`pipeio_run(..., snakemake_cmd=
   ".pixi/envs/default/bin/snakemake")`) for projects that pin
   differently.

Option (1) is the default fix; option (2) is a back-pocket escape.

### Related

- Wildcards passed via `pipeio_run(wildcards={...})` go through as
  `--filter-subject` etc., which exist in `snakebids` CLI but **not**
  plain `snakemake` CLI. Once (1) is fixed, this will also work; until
  then the workaround is to pass `targets=[resolved_path]`.
- The `EXIT_CODE=0` rubber-stamp issue filed 2026-04-21 is what lets
  this failure go unnoticed. Fixing that tool-side would surface this
  class of env-resolution bug immediately.

### Priority

High — blocks scheduled-agent pipeline work across all pixi-migrated
projects (gecog + pixecog + msol). Each failed run wastes ~10-20 min of
opus session time with `exit_code=0` masking the real failure.
