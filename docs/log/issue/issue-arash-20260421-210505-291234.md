## `pipeio_run` reports `EXIT_CODE=0` despite rule failures

### Symptom

`pipeio_run_status` returns `status: "ok"` and `exit_code: 0` even when
Snakemake logged `WorkflowError: At least one job did not complete successfully.`

Repro in gecog (2026-04-20 / 2026-04-21):

- Run `a31f5057` (spectrogram): `whiten` rule crashed on an xarray â†’
  statsmodels incompatibility; Snakemake exited with `WorkflowError`; no
  output files; log ends with `EXIT_CODE=0`. `pipeio_run_status` reported
  `status: ok`.
- Run `fbb75d17` (preprocess_ieeg): 4 `bandpass` rules failed; `status: ok`
  exit=0. Only visible on inspection that `bandfilter/` dir was empty.

### Impact

- Schedulers / agents trust exit_code=0 as "done" â†’ rubber-stamped
  completions (observed twice in scheduled batches in gecog).
- `pipeio_run_status.failed_rules` is populated correctly, but the
  top-level `exit_code` + `status` are misleading.

### Suspected cause

The shell wrapper that launches snakemake inside the detached screen session
likely emits `echo EXIT_CODE=$?` after `set +e` (or similar), so the
non-zero snakemake exit gets swallowed. Alternatively, `--keep-going` may
return 0 from snakemake when any job completes, even if others failed.

### Suggested fix

Either:
1. Respect snakemake's real exit code â€” propagate it through the screen
   wrapper without masking.
2. Compute `status` in `pipeio_run_status` from `failed_rules` / `WorkflowError`
   in log rather than trusting the shell exit.

Option 2 is safer (doesn't require changing launch plumbing).

### Evidence

- `code/pipelines/spectrogram/.snakemake/log/run-20260421-195506-a31f5057.log`
  (gecog) â€” final lines show `WorkflowError: ... EXIT_CODE=0` back-to-back.
- `code/pipelines/preprocess_ieeg/.snakemake/log/run-20260420-192004-fbb75d17.log`
  â€” same pattern.

### Memory note

There's already a memory entry in gecog
(`feedback_snakemake_venv.md` and related) noting "scheduled tasks
reporting exit_code=0 without doing work". This bug is the root-cause of
that class of failures.
