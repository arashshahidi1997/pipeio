# Proposal: Progressive, Spec-Centered Workflows for Pipeio

## Status

Draft proposal for architecture and workflow design.

## Purpose

This proposal outlines a workflow model for **pipeio** based on the design discussions so far. The core shift is to treat **declarative specs as the primary authoring surface for workflow structure**, while notebooks, scripts, Snakemake rules, configs, and docs become synchronized realizations of that structure.

The goal is not to force all users into a single path. Instead, pipeio should support **multiple sanctioned entry points** and provide explicit reconciliation steps so exploratory work can progressively formalize into robust pipeline infrastructure.

---

## 1. Motivation

Pipeio sits in a distinct role within the projio ecosystem:

* **indexio, codio, biblio, notio** organize knowledge
* **pipeio** organizes execution

In research repositories, especially notebook-heavy scientific codebases, the difficult part is rarely just writing Python or Snakemake syntax. The difficult part is deciding:

* what the computational unit is
* what it consumes
* what it produces
* which parameters matter
* which artifacts are exploratory versus operational
* how outputs should be named, validated, documented, and discovered
* how humans and AI agents should interact with the workflow system

These decisions are typically scattered across:

* notebook cells
* path strings
* Snakefiles
* config files
* docs
* naming conventions
* lab folklore

Pipeio should centralize those decisions into explicit, machine-readable artifacts.

---

## 2. Core Thesis

### Spec writing is the new code

For workflow structure, the primary act of programming should be **declaring interfaces and intent**, not manually reproducing the same structural information across multiple files.

This does **not** mean specs replace algorithmic implementation. Rather:

* specs replace **structural boilerplate**
* specs constrain and generate **coordination artifacts**
* code remains the place for **algorithmic logic**
* reconciliation ensures specs and implementation remain aligned

A better precise formulation is:

> In pipeio, declarative specs are the primary programming surface for workflow structure; notebooks, scripts, rules, configs, and docs are synchronized implementations of those specs.

---

## 3. Architectural Principle: The Mod as the Central Contract

The strongest emerging design is to make the **mod** the core declarative object.

### Proposed interpretation

* **Pipe** = scientific domain or processing stage
* **Flow** = concrete workflow container
* **Mod** = declarative contract for one logical computational unit

A mod should not just be a label derived from rule prefixes. It should become the semantic center that binds together:

* notebook
* script
* Snakemake rule(s)
* config references
* input/output contracts
* docs
* validation
* agent reasoning

### Conceptual role of the mod

A mod spec should define, at minimum:

* identity
* summary/purpose
* entities
* expected inputs
* expected outputs
* parameters
* linked implementation artifacts
* validation hints
* maturity/status

So instead of thinking:

* notebook first, then promote to rules

pipeio should support:

* mod spec first or early
* notebook and script evolve against the spec
* Snakemake rules and config are scaffolded from the spec
* notebook analysis reconciles reality back into the spec

This turns notebook analysis from **reverse engineering** into **comparison against a declared contract**.

---

## 4. Key Design Shift: From Promotion to Reconciliation

The original notebook-promotion idea is valuable, but it should be reframed.

### Avoid this framing

* “turn notebook into pipeline”

That overpromises and encourages brittle inference.

### Prefer this framing

* notebook provides evidence
* mod spec declares intent
* pipeio reconciles evidence and intent
* pipeio scaffolds operational artifacts from the reconciled spec

This leads to a better lifecycle:

1. draft a spec
2. explore in notebook and/or script
3. analyze implementation artifacts
4. propose spec updates or drift warnings
5. confirm
6. generate/update operational artifacts
7. validate and run

This makes the system safer, clearer, and more trustworthy.

---

## 5. Proposed Artifact Roles

### 5.1 Mod spec

The canonical declarative object for one computational unit.

Should define:

* what this unit is
* what it consumes
* what it produces
* which parameters it exposes
* which entities it binds
* which notebook/script/rule belong to it

### 5.2 Notebook

Exploratory or diagnostic realization of the mod.

Useful for:

* prototyping
* parameter tuning
* visualization
* method exploration
* sanity checks
* narrative explanation

Notebook should be allowed to be incomplete or messy during early phases.

### 5.3 Script/module

Operational realization of the mod.

Useful for:

* reproducible batch execution
* clean callable entrypoints
* testability
* rule integration

### 5.4 Snakemake rule

Scheduling and orchestration wrapper.

Useful for:

* dependency graph integration
* parallelization
* resource declarations
* workflow-level reproducibility

### 5.5 Flow config

Flow-level path and product contract substrate.

Useful for:

* flow-scoped directories
* named output groups
* parameter defaults
* resolver configuration

### 5.6 Contracts

Validation layer.

Useful for:

* existence and non-emptiness checks
* shape/schema checks
* required fields/columns
* post-run integrity

## 6. `pipeio_nb_analyze`: Recommended Role

Notebook analysis should remain important, but its scope should be carefully defined.

## Recommendation

`pipeio_nb_analyze` should be primarily **deterministic structural analysis**, not fully semantic interpretation.

### Deterministic responsibilities

* imports
* top-level definitions
* known API calls
* file read/write patterns when explicit
* parameter declarations
* section structure
* plotting sections
* presence of `PipelineContext`
* notebook metadata and linked artifacts

### Output should include

* candidate inputs
* candidate outputs
* candidate parameters
* side-effectful sections
* ambiguities
* likely diagnostics-only cells
* possible extraction units
* drift relative to mod spec

### Optional second layer

A later AI-assisted layer can interpret the structural output and propose:

* rule boundaries
* script extraction suggestions
* missing parameters
* better documentation
* possible mod splits

But structural truth should remain deterministic and testable.

---

## 7. `PipelineContext`: Recommended Role

`PipelineContext` adds real value, but it must stay ergonomic.

Its purpose is not just path shortening. Its real purpose is to provide a **typed access layer over declared pipeline products**.

### Benefits

* centralizes naming conventions
* reduces path duplication
* makes products explicit
* improves agent reasoning
* supports multiple resolver backends
* allows existence/expansion checks at the product level

### Recommendation

Provide both:

#### Simple API

Best for notebooks and quick scripts:

* `ctx.path(...)`
* `ctx.have(...)`
* `ctx.expand(...)`

#### Fluent API

Best for richer and more readable workflows:

* `ctx.session(...).stage(...).path(...)`

The simple API should remain canonical for ergonomics.

## 8. Workflow Principle: Progressive Formalization

Pipeio should support a model of **progressive formalization**.

Users do not always begin with the same artifact. Sometimes they start with an idea, sometimes a notebook, sometimes an existing rule, sometimes legacy code. Pipeio should not impose a single rigid path.

Instead, the system should support multiple entry points that converge on the same mature state.

### Common maturity progression

#### Stage 0 — Idea

The user knows only:

* scientific goal
* rough inputs
* rough desired outputs

#### Stage 1 — Draft structure

A draft mod spec and/or notebook is created.

#### Stage 2 — Exploratory realization

Notebook and/or script evolves.

#### Stage 3 — Reconciliation

Pipeio compares implementation with draft structure.

#### Stage 4 — Operationalization

Pipeio scaffolds or patches:

* scripts
* rules
* config
* contracts
* docs

#### Stage 5 — Locked module

Validated, documented, runnable.

---

## 9. Proposed Supported Workflow Families

Pipeio should explicitly support at least six alternative workflows.

---

## 9.1 Spec-first workflow

### Use when

* a new method is being introduced
* the user already knows the rough computational boundary
* structured agent assistance is desired
* outputs and parameters are partly understood early

### Flow

`idea -> draft mod spec -> notebook/script/rule scaffolds -> exploration -> reconciliation -> confirmation -> operationalization`

### Example sequence

* create draft mod spec
* create notebook from mod
* create script stub from mod
* create rule stub from mod
* explore in notebook
* analyze notebook
* reconcile notebook evidence with spec
* confirm spec updates
* generate/update config and contracts
* run

### Strengths

* strong structure from the beginning
* high legibility for agents
* reduces reverse engineering later
* makes products visible early

### Weaknesses

* may feel too formal for very early exploration
* draft spec may be incomplete or wrong

### Fallback

Allow an intentionally lightweight draft spec with only:

* name
* summary
* intended inputs
* intended outputs
* rough params

---

## 9.2 Notebook-first workflow

### Use when

* the user is exploring a new idea
* method boundaries are unclear
* rapid scientific iteration is the priority
* a notebook already exists

### Flow

`idea -> notebook -> notebook analysis -> draft mod extraction -> review -> scaffolding -> operationalization`

### Example sequence

* create or import notebook
* perform exploratory work
* run notebook analysis
* extract or draft a mod spec from evidence
* review candidate inputs/outputs/params
* generate script and rule scaffolds
* patch config/contracts
* validate and run

### Strengths

* matches normal research practice
* low friction
* natural for uncertain methods

### Weaknesses

* notebook may be messy
* promotion is more heuristic
* hidden state and ambiguity are more likely

### Fallback

If analysis confidence is low, only produce:

* observations
* candidate params
* candidate inputs/outputs
* drift warnings

Do not overgenerate.

---

## 9.3 Rule-first / pipeline-first workflow

### Use when

* the user already knows how the pipeline step should run
* productionization matters more than exploration
* the code already exists in script form
* a notebook is optional

### Flow

`idea -> mod spec and rule/script scaffolds -> direct script implementation -> optional notebook -> validation`

### Strengths

* fast path to production
* suitable for experienced pipeline developers
* avoids dependence on notebooks

### Weaknesses

* may underdocument exploratory rationale
* notebook/docs can lag behind

### Fallback

Require only:

* mod spec
* script entrypoint
* declared outputs

Notebook can remain optional.

---

## 9.4 Config-first / contract-first workflow

### Use when

* repository standardization is the main goal
* outputs and naming conventions must be stabilized early
* downstream tooling depends on a product registry
* multi-user coordination is important

### Flow

`idea -> flow config / output registry / contracts -> mod binding -> notebook/script/rules later`

### Strengths

* stabilizes interfaces early
* good for repo-wide cleanup
* strong reproducibility and discoverability

### Weaknesses

* can feel abstract before a method exists
* may discourage fast experimentation if too rigid

### Fallback

Support draft contracts and provisional output groups.

---

## 9.5 Clone/adapt-existing-mod workflow

### Use when

* a new method is a variant of an existing one
* the user wants to adapt a prior analysis pattern
* structure reuse is more important than clean-sheet design

### Flow

`existing mod -> clone spec -> clone or regenerate artifacts -> modify -> reconcile`

### Strengths

* likely one of the fastest real-world workflows
* leverages working patterns
* ideal for agent assistance

### Weaknesses

* may carry forward legacy design flaws
* risks structural drift if copied too casually

### Fallback

Support both:

* full clone
* clean clone from spec only

---

## 9.6 Minimal/manual fallback workflow

### Use when

* the task is weird or one-off
* the repo is legacy
* the user wants complete manual control
* pipeio cannot infer enough safely

### Flow

`manual notebook/script/rule/config work -> later registration or scan into pipeio`

### Strengths

* avoids blocking users
* realistic for migration and legacy adoption
* allows gradual adoption

### Weaknesses

* least consistent
* most drift-prone
* weakest for agents

### Role

This should remain a sanctioned fallback, not a failure case.

---

## 10. Recommended Default Workflow

Pipeio should still present one preferred path for new users.

### Recommended default

`idea -> draft mod spec -> notebook from mod -> exploration -> notebook analysis -> reconciliation -> confirmation -> script/rule/config sync -> validation -> run`

This path balances:

* early structure
* notebook-centric research practice
* controlled automation
* safe operationalization

### Why this default is strong

* it preserves notebook exploration
* it makes the mod spec central
* it introduces a review gate before generation
* it avoids claiming that notebooks can be cleanly compiled into workflows

---

## 11. Confirmation and Reconciliation Model

The reconciliation step should be explicit and reviewable.

### Recommendation

Pipeio should distinguish between:

#### Observations

Facts extracted from notebook/script/rule/config.

#### Proposals

Suggested updates to:

* mod spec
* parameters
* outputs
* docs
* contracts
* rule scaffold

#### Confirmed changes

Changes applied after human or policy-controlled agent approval.

### Confirmation modes

#### Human confirm

Best default.

#### Policy-limited agent confirm

Allowed for safe additive updates such as:

* docs
* minor metadata
* non-breaking params
* scaffold refreshes

#### Draft patch output

Instead of mutating directly, pipeio can write:

* reconciliation plan
* patch file
* promotion plan

This is an excellent fallback.

---

## 12. Generation Responsibilities

Generation should remain disciplined.

### Mod spec should generate

* notebook scaffold
* script scaffold
* rule scaffold
* docs stub
* contract stub

### Notebook analysis should generate

* observations
* reconciliation proposal
* drift report
* candidate params
* candidate I/O updates

### Rule parsing should generate

* implementation inventory
* rule signature view
* drift report relative to spec

### Config should support

* flow-level declarations
* output groups
* defaults
* resolver integration

Config should not become the sole semantic source for mod meaning.

## 13. Recommended Command/Tool Direction

The current tool surface is promising, but should be organized around the workflow model above.

### Good low-level building blocks

* mod create
* nb create
* nb analyze
* rule stub
* config patch
* contracts validate
* run
* registry scan/validate

### Recommended additions or refinements

* `pipeio_mod_create --draft`
* `pipeio_nb_create --from-mod`
* `pipeio_script_stub --from-mod`
* `pipeio_rule_stub --from-mod`
* `pipeio_mod_reconcile`
* `pipeio_mod_confirm`
* `pipeio_mod_clone`
* `pipeio_mod_extract --from-notebook`
* `pipeio_mod_operationalize`

### Important distinction

Tooling should clearly distinguish:

* read-only discovery
* scaffold/generation
* execution/mutation

This is especially important for agent permissions.

---

## 14. Practical Design Constraints

To succeed, the system must avoid several failure modes.

### 14.1 Overloaded specs

If specs become too verbose, users will bypass them.

### 14.2 Magical notebook inference

If notebook analysis overclaims semantic understanding, trust will collapse.

### 14.3 Ceremony in notebooks

If simple actions require too much abstraction, users will revert to raw paths.

### 14.4 Hidden authority

Users must know which file is the source of truth.

### 14.5 Forced workflow rigidity

Pipeio must support multiple legitimate entry points.

---

## 15. Explicit Source-of-Truth Model

The architecture should clearly state the role of each artifact.

### Proposed source-of-truth division

* **Registry**: topology and locations
* **Flow config**: flow-level paths, output groups, defaults
* **Mod spec**: semantic contract for one computational unit
* **Snakefile/rules**: execution graph and scheduling
* **Notebook**: exploratory or diagnostic implementation
* **Script/module**: operational implementation
* **Contracts**: validation expectations
* **Docs**: user-facing explanation and navigation

This separation should be written directly into pipeio docs.

---

## 16. Summary Recommendation

Pipeio should adopt a **progressive, spec-centered workflow architecture** in which the mod becomes the central declarative contract for each computational unit.

The recommended model is:

* treat specs as primary workflow-structure code
* treat notebooks, scripts, rules, configs, and docs as synchronized realizations
* support multiple entry workflows
* make reconciliation explicit
* keep notebook analysis deterministic at its core
* preserve a lightweight manual fallback
* optimize for both researchers and agentic workflows

The most important practical shift is this:

> Pipeio should move from a notebook-promotion story to a spec-centered reconciliation story.

That gives the system a stronger architecture, safer automation, clearer agent affordances, and a more realistic fit to research practice.
